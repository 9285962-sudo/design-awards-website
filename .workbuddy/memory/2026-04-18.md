## 文章发布流程踩坑总结（全天复盘）

### 核心问题：脚本设计缺陷
**根本原因**：`build-articles.js` 脚本只负责生成 HTML，不自动更新 `data/news.json`，导致两端数据不同步。

### 具体错误
1. **路径混乱**：
   - 文章生成在 `public/articles/`，但页面链接指向 `/news/articles/`
   - 修复：统一改为 `/articles/`

2. **数据不同步**：
   - 新增 markdown 文件 → 脚本生成 HTML → 但 `news.json` 未更新 → 前端列表不显示
   - 删除 markdown 文件 → HTML 和 `news.json` 记录残留 → 前端显示已删文章

3. **页面样式不一致**：
   - `events.js` 使用 `auto-fit`，其他页面用 `auto-fill`，导致卡片宽度不一致
   - 摘要文字未限制行数，显示过长

4. **硬编码数据**：
   - `notice.js` 使用静态数组，不从 `news.json` 读取
   - 导致新增文章无法显示在报名通知分类

### 教训
1. **脚本应该原子化**：一个脚本完成"扫描-生成-更新"完整流程，而不是分步手动操作
2. **数据单一来源**：所有页面必须从同一个数据源（`news.json`）读取，不能硬编码
3. **先清空再生成**：脚本运行前应先清理旧文件，避免残留数据干扰
4. **样式统一**：同类页面必须复用相同样式，不能各自维护

### 最终方案
- 重写 `build-articles.js`：每次运行时清空 `public/articles/` 和 `data/news.json`，重新扫描 `content/` 目录生成全部内容
- 统一三个子页面（notice/winners/events）从 `news.json` 读取数据
- 统一卡片样式和摘要行数限制

### 发布流程（最终版）
1. 把 markdown 文件放到 `content/` 目录
2. 运行 `node scripts/build-articles.js`（自动处理所有事情）
3. 运行 `npm run build && npx wrangler pages deploy out --branch=main`

**关键认知**：文章发布的核心不是生成 HTML，而是维护 `data/news.json` 这个索引文件。前端列表页完全依赖它，HTML 只是详情页内容。
